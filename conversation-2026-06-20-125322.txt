  项目审核报告:Statistics Visualization Platform

  审核范围:13 个独立 Vite 应用 + 平台 shell,部署在 GitHub Pages 的纯静态教学可视
  化平台。下面按你要求的五个维度逐项给出批评,每条都附带具体文件/行号佐证。

  ---
  一、逻辑错误(优先级最高)

  1.1 置信区间宽度公式漏除 √n —— 教学上的硬伤

  apps/confidence-interval/src/components/confidence-interval/model.ts:71-74:
  const z = zScore(confidenceLevel);
  const margin = z * stddev / Math.sqrt(samples.length);
  这一处是对的。但是 MES 模板
  apps/simulation-mcmc/src/simulation/engine.ts:546-566 的 confidenceInterval()
  函数却写成:
  const width = num(controls, "nSigma", 2) * Number(sigma.toFixed(2));
  宽度 = z · σ,而不是 z · σ / 
  √n。同仓库两个模块对"置信区间宽度"给出相互矛盾的公式,而 sampleSize
  这个控件还被显式画在 UI 上却完全不参与计算。一个教统计学的平台在自己的"置信区
  间"模块里写错置信区间公式,这种问题应该当场打回。

  1.2 二项分布 PMF 在 x > n 时给出非零值

  apps/simulation-mcmc/src/simulation/engine.ts:583-588 计算 C(size, x) · p^x · 
  (1-p)^(size-x):
  const size = Math.max(1, Math.round(b));
  const choose = Array.from({ length: x }, (_, index) => (size - index) / (index
  + 1))
                    .reduce((product, value) => product * value, 1);
  xs 的范围是 0..Math.max(8, b+1),所以 x 可能等于 size+1。数学上 C(n, n+1) = 
  0,但这里连乘会得到一个非零(甚至为负,当 size - index < 0 时)的值,把 PMF
  画成"超过 n 还有概率"的错误曲线。

  1.3 Box-Muller 在 u₁ = 0 时返回 -∞

  apps/confidence-interval/src/components/confidence-interval/model.ts:40-44:
  const u = Math.random();
  const v = Math.random();
  const z = Math.sqrt(-2.0 * Math.log(u)) * Math.cos(2.0 * Math.PI * v);
  没有 Math.max(u, EPSILON) 保护。其他模块(apps/*/src/utils/random.ts 中的
  normalRandom)都做了保护,这一处漏掉。Math.random() 返回 0 的概率虽小但不为
  0,且一旦发生会污染整次模拟结果(出现 NaN/Infinity 拖低覆盖率)。

  1.4 置信区间模块用未播种 Math.random(),违反"可复现"教学承诺

  同上文件,所有样本都来自 Math.random()。这导致:
  - 每次刷新覆盖率指标都在漂移,课堂上"95% 置信区间应该覆盖 95%"的演示可能展示成
  88% 或 99%。
  - 与 WALS Simulation 系列(全用 createRandom(seed))行为不一致,平台内部对 RNG
  的处理口径不统一。

  1.5 Buffon 投针在 planeWidth < needleLength 时公式失效

  engine.ts:106-136,needleLength = 1 硬编码,而控件允许 planeWidth = 1(控件只设了
  Math.max(1, ...))。经典 Buffon 公式要求 L ≤ d,在边界上估计发散,UI
  却照样给出一个看起来合理的数字,误导学生。

  1.6 jStat 与手写 erf 同时存在

  apps/type-error/src/components/TypeErrorsApp/model.ts:5 引入
  jstat,但同一文件第 33-49 行手写了 Abramowitz-Stegun 近似 erf。jStat 已经提供
  erf/normalCDF/normal.inv——既然为了精确分位数引了 jStat,为什么 PDF/CDF
  又用手写近似?两套实现精度不一致,且增加 ~70KB 包体。

  1.7 i18n.ts 反向字典丢词

  apps/shared/i18n.ts:538-544 用 ??= 反转 enToZh。但 enToZh
  里多个英文短语映射到同一中文(例如 "sample size" 和 "Sample Size"
  都映射到"样本量"),反转后只有第一个英文能被还原。一旦 UI
  上展示了"样本量",切回英文可能拿到 "sample size" 而非 "Sample
  Size"——大小写丢失。这是把"双向字典"伪装成正反两字典的经典坑。

  1.8 Math.min/max(...values) 在大数组上会栈溢出

  apps/*/src/simulation/engine.ts 的 histogram() 函数和 bootstrapMax() 都用了
  Math.min(...values) / Math.max(...values)。V8 的展开参数上限约
  65535(bootstrapMax 在 replicates=1000、values.length
  取决于用户输入字符串时仍安全,但用户可以把 data
  字符串填到几万元素)。meanBootstrap 的 sample 默认 10000,临界。一旦超阈值直接
  RangeError,而不是优雅降级——同样的 bug 在 13 个 app 里都被复制了一份(md5sum
  完全一致)。

  1.9 城市数据中文键"planing"拼写错误

  apps/mes-linear-regression/src/components/MesLinearRegressionApp/module-config
  .ts:154,160,166,... 把 planning 写成 planing,然后 engine.ts:612, 626 也用
  city.planing。从教学角度学生看到的是"已规划数",但变量名是动词"飞行"。这种 typo
  沿着数据 → 引擎 → 表格列名蔓延,但因为是"行为保留"重构规则,没人敢动。

  1.10 教材数据集的来源与单位没有 source 字段

  apps/regression/public/data/positive-correlation.json 等六个合成数据集只有
  id/name/data,没有来源/单位/采集时间。但 chapter-8-height-father-height.json
  标了 source: "表8-9",却没说采自哪本教材、哪个学校、哪个年份。一个用于教学的数
  据集,如果学生问"这数据是真的吗",目前无法溯源。

  二、设计思路

  2.1 用 iframe 嵌入 13 个独立 Vite 应用,代价巨大

  src/main.ts:532-538 通过 iframe 加载每个模块。结果是:
  - 每次切换模块都要重新解析 HTML、加载 JS、初始化 d3 + Cycle + xstream,冷启动 
  200-800ms。
  - 13 个模块的 vendor
  chunk(d3、Cycle、xstream)在浏览器缓存层面虽然能命中,但每个 iframe 还要独立构建
  DOM/Cycle context。
  - iframe 与父窗口的语言同步靠 postMessage(src/main.ts:241-256)——一旦 message
  时序错乱,模块会短暂显示错误语言。

  为什么不用单一的 monorepo + 路由懒加载? 唯一站得住的理由是"每个 app
  都有独立上游仓库"(README 的"integration
  notes"暗示了这点),但既然已经全部代码都搬进了 apps/,iframe 隔离就只剩坏处。

  2.2 injectModuleResize 是危险的外科手术

  src/main.ts:291-471 在 iframe 加载完成后,从父窗口注入 CSS 和 pointer 
  事件处理进 iframe 文档:
  const layout = frameDocument.querySelector<HTMLElement>(".module-layout");
  // ...
  style.textContent = `... 100 行 CSS with !important ...`;
  这意味着:
  - 平台 shell 与每个 app 的 DOM
  结构强耦合(.module-layout、.experiment-board、.teaching-area),任何 app 改
  class 名都会让分隔条功能失效。
  - 大量 !important
  覆盖,本质是用父窗口"治理"子窗口样式,反过来又限制子窗口自己改样式。
  - MutationObserver 在 layout 上监听子节点变化,只要 app 内部重渲染就可能触发
  injectModuleResize 重新跑(src/main.ts:463-470)。

  这等于把"响应式布局"问题用胶布贴在 iframe 边界上。如果改成单 SPA + CSS
  Grid,这些都是不必要的。

  2.3 i18n 用"全量英中字典 + 动态正则"

  apps/shared/i18n.ts 的设计是:
  1. 维护一个 ~530 行的 enToZh 字典。
  2. 反转出 zhToEn。
  3. 还有一组正则匹配 target 1.234、5 burn-in draws
  这类带数字的模板字符串(localizeDynamicText)。

  问题:
  - 字典是字符串到字符串的硬编码,没有 key,任何文案微调都要同步改字典,测试
  template-zh-completeness.test.ts 反映出团队已经在被这件事折磨。
  - 新增一个 app 就要往字典塞 50-100 行,字典只会越长越失控。
  - 没有 ICU
  MessageFormat、没有复数/性别、没有占位符抽象——copy.samples(sampleCount) 这种是
  Cycle 视图里手写的函数,绕过了 i18n 系统。

  应当替换成 @formatjs/intl 或简单的 key → { zh, en } 表。

  2.4 Cycle.js + xstream 是 2018 年的技术选型,且团队明显不熟

  apps/regression/src/App/model.ts:51-68、createTempPixels、createFinalPixels、c
  reateIsDragging、createIsShowCustomLine——这一组 stream 操作里 .map((data: any)
  => data) 出现了两次,是 stream 内容的不必要透传,典型"在学 Cycle
  时写的代码"。所有 13 个 app 都依赖 xstream@11(2019
  年最后更新)、@cycle/dom@23,生态停滞。新成员上手成本极高。

  2.5 部署用 cancel-in-progress: false

  .github/workflows/deploy.yml:14-16:
  concurrency:
    group: pages
    cancel-in-progress: false
  group: pages 是 GitHub Pages 的全局共享组,跨分支、跨仓库(如果都部署到
  Pages)排队。这意味着两个无关的项目部署会互相阻塞。应改成 group: pages-${{ 
  github.ref }}。

  三、数据来源

  3.1 内置数据集版权与可溯源性不明

  - apps/regression/public/data/positive-correlation.json 等 6
  个"合成数据集"没有 source 字段,无法判断是人造的还是真实采样。
  - chapter-8-*.json 系列标注了"表8-9"但没有教材名称、作者、ISBN、出版年份。docs
  /integration/regression-source-comparison.md 也只对比了"平台 app vs 私有
  regression-visualizer vs 公开 code repo",没有触及数据本身的可溯源性。
  - 摩天楼数据集(module-config.ts:11-174)列了 24 座中国城市的
  GDP、已建摩天楼数、计划摩天楼数,没有任何年份、口径、来源标注。"上海 16872.42
  亿 GDP"是哪一年的?这种数据放在教学平台上,学生无法验证。

  3.2 控件默认值与数据规模不匹配,可能误导

  apps/simulation-mcmc/src/simulation/engine.ts:499-517 中 ANOVA 的 fuel 和
  temperature 数据集是硬编码常量,且数值精度(如
  8.65、263、254)像是被截断过的真实数据。random 数据集是 normalRandom,使用同一个
  rng = createRandom(seed) 但 seed 默认 510 ——
  所有学生看到的"随机数据"都完全一致,无法演示抽样变异,违反 ANOVA 教学初衷。

  3.3 confidence-interval 用 Math.random() 而 WALS 系列用 seed

  跨模块数据生成口径不一致:同一个平台,核心模块用真随机,模板模块用伪随机。学生在
  两个模块间切换会困惑"为什么这个能复现那个不能"。

  3.4 parseNumberList 没有输入校验

  apps/simulation-mcmc/src/utils/format.ts(13 个 app 共享同一份)。bootstrapMax
  中 parseNumberList(str(controls, "data", "0.5,0.6,0.7"))
  直接把字符串拆数字。用户输入 1, abc, 2 会得到 [1, NaN, 2],然后 Math.max(...[1,
  NaN, 2]) 返回 NaN,整列 bootstrap 输出全是
  NaN。教学场景下学生乱输字符串是必然事件。

  四、模块化

  4.1 13 份完全相同的 random.ts 和 format.ts

  md5sum apps/*/src/utils/random.ts   → 全部 64522855...
  md5sum apps/*/src/utils/format.ts   → 全部 fe81b9ca...
  而 apps/shared/ 目录已经存在,却只放了 i18n.ts 和 language.ts。所有跨 app
  共享的统计工具(RNG、格式化、mean、variance、quantile)被复制了 13 份。修复
  Math.min(...values) 这种 bug 要改 13 个文件。

  4.2 9 个 WALS app 的 index.ts/view.ts 完全字节相同

  md5sum apps/*/src/components/*/index.ts → f1a64384551d9378e1a97072d4a189fa × 9
  md5sum apps/*/src/components/*/view.ts  → 4e733a975dda2c797497e93347716473 × 9
  9 个 MCMC/CLT/Resampling/MES app 共享同一份 view.ts(模板渲染器)、同一份
  index.ts(Cycle 入口)。但每个 app 都有自己独立的
  package.json、node_modules(~14MB × 11 = 154MB)、独立的 vite.config、独立的
  tsconfig。

  正确做法:把 9 个 app 合并成一个 apps/wals-template/,通过 module-config.ts
  的差异驱动内容,template 工厂根据 kind 选择 runExample
  的分支。当前架构等于把"配置驱动"硬塞回"复制粘贴"。

  4.3 跨 workspace 相对路径逃逸

  apps/mes-anova/src/components/MesAnovaApp/index.ts:1:
  import { languageStream } from "../../../../shared/language";
  四次 .. 跳出当前 workspace 包,直奔兄弟目录 apps/shared/。npm workspaces
  不应这样跨包——shared 没有作为依赖被任何 package.json 声明,Vite 能解析只是因为
  Node 的目录上溯,既不在 dependencies 里也没有版本约束。一次 npm install
  重新链接就可能断掉。

  正确做法:把 apps/shared 注册成 workspace 包,在子包 package.json 里写
  "@statistics-platform/shared": "*",然后 import { languageStream } from 
  "@statistics-platform/shared"。

  4.4 engine.ts 在 10 个 app 中是不同文件但 80% 内容相同

  md5sum apps/*/src/simulation/engine.ts → 10 个不同的 md5
  但内容上,piCircle、buffon、bootstrapMax、mcmcMixture、politician、anova、confi
  denceInterval、distribution、linearRegressionExample 等函数在每个 engine.ts 
  里都被复制粘贴,只是不同 app 用到不同的子集。这意味着 confidenceInterval 那个
  √n 公式 bug 存在于 10 份代码中。

  4.5 顶层 package.json 与子包脚本重复

  scripts/build.mjs 维护了一个 13 个 app 的硬编码数组(apps 列表),且与
  src/visualizers.ts 中的 visualizers 数组双源。新增 app 要同时改这两处,以及
  package.json 的 workspaces: ["apps/*"]——三处一致性没有测试守护。

  五、高并发与性能

  5.1 主线程同步模拟,且每次 slider 变动都重跑全量模拟

  apps/simulation-mcmc/src/components/SimulationMcmcApp/model.ts:64-76:
  const updateReducer$ = actions.updateControl$.map(
    ({ id, value }): Reducer =>
      (state) => createState(state.activeExample.id, { ...state.controls, [id]:
  value }, state.seed, state.language)
  );
  createState 调用 runExample,后者跑完整的 piCircle/mcmcMixture/politician
  等模拟。没有 debounce、没有 requestAnimationFrame、没有 Web Worker。grep -rn 
  "Worker\|debounce\|requestAnimationFrame" apps/*/src 全空。

  后果:用户拖动 steps slider 从 1000 → 20000,会触发几十次连续模拟,每次主线程冻结
  50-300ms。politician 在 20000 步 × 接受率计算下,单次模拟 ~150ms,slider
  拖动期间浏览器完全无响应。

  5.2 13 个独立 vendor chunk,毫无共享

  apps/regression/vite.config.ts:24-30 把所有 node_modules 单独打到
  vendor.js。每个 app 都这样做,但因为 iframe 隔离 + 不同 app 路径前缀不同,GitHub
  Pages 的浏览器缓存命中率有限。首次切换到一个新 app 仍然要下载 ~600KB 的 d3 +
  Cycle + xstream。

  13 个 app × ~14MB node_modules × npm ci 全量安装 → CI 构建时间膨胀。

  5.3 构建串行

  scripts/build.mjs:96-111:
  for (const app of apps) {
    run("npm", ["run", "build", ...], app.cwd);
  }
  13 个 app 用 execFileSync 串行构建,每个都冷启动一个 Vite + tsc。GitHub Actions
  上整体构建时间能从 ~3 分钟降到 ~30 秒(Promise.all 并发,只要 13 个 outDir
  不冲突——它们已经在 dist/apps/<name>/ 隔离)。

  5.4 平台 shell 也会触发"全量重渲染"

  src/main.ts:170-239 的 syncActiveVisualizer 在每次 hashchange 时:

  ## 一、逻辑错误                                                               
  ### 1. Box-Muller 变换浪费 50% 随机性                                         
  apps/confidence-interval/src/components/confidence-interval/model.ts          
                                                                                
  Box-Muller 一次产生两个独立正态值（cos + sin），此处只取了 cos 分量，sin      
  分量被丢弃。每次调用多浪费一次 Math.random()                                  
  和一次对数运算。正确做法是返回两个值并缓存第二个。                            
                                                                                
  ### 2. erf / normalCDF 重复实现且已依赖 jstat                                 
  apps/type-error/src/components/TypeErrorsApp/model.ts 手写了 erf() 和         
  normalCDF() （约 20 行），但同文件已经 import jStat from 'jstat' 。jstat      
  本身提供了 jStat.erf() 和 jStat.normal.cdf()                                  
  ，不仅更可靠，而且少维护代码。同样的问题在 simulation-introduction 、         
  simulation-clt 的 engine.ts 中分别重复出现。                                  
                                                                                
  ### 3. 置信区间使用总体标准差而非样本标准误                                   
  apps/confidence-interval/src/components/confidence-interval/model.ts          
                                                                                
  这里的 stddev                                                                 
  是用户在控件中设置的"总体标准差"。在真实统计教学中，更常见的场景是            
  未知总体标准差、用样本标准差 s 替代 （此时应用 t 分布而非正态 z 值）。当前实  
  现只覆盖了"已知总体方差"这一特殊情形，且没有任何文字说明这个假设前提。        
                                                                                
  ### 4. zScore 硬编码 fallback 无保护                                          
  apps/confidence-interval/src/components/confidence-interval/model.ts          
                                                                                
  浮点数严格相等 === 本身就是风险的（ 0.95 可能被存储为 0.9499999... ），且     
  fallback 无任何 console.warn，静默给出错误结果。应该使用 jstat 的             
  jStat.normal.inv(1 - (1 - level) / 2, 0, 1) 来通用计算。                      
                                                                                
  ### 5. histogram 函数除零隐患                                                 
  多个 engine.ts 中：                                                           
                                                                                
  当所有值相等时 (max === min)， || 1 兜底防止除零，但此时所有点落入            
  bin[0]，直方图显示可能产生误导（看起来像均匀分布）。
